# DHIS2 Map MART Behavior

DHIS2 Map charts use two different local sources on purpose:

- metric data and organisation-unit hierarchy come from the MART dataset query
  path, just like other Superset charts
- boundary GeoJSON stays on the staged local DHIS2 boundary snapshot path

The map must not move boundary loading onto the MART tables. The staged
boundary snapshots are the fast local source for geometry and are kept separate
from the MART facts.

## Problem

Older MART datasets could exist with hierarchy columns such as `national`,
`region`, and `district_city` marked as `dhis2_is_ou_hierarchy`, but without
the `dhis2_ou_level` metadata that newer builds attach. Some datasets also had
helper columns such as `period_variant` mis-tagged as hierarchy columns.

That broke two important behaviors:

1. Explore could not reconstruct `dhis2_hierarchy_columns` from datasource
   metadata, so DHIS2 Map requests dropped back to incomplete hierarchy form
   data.
2. Backend staged-hierarchy predicates could not identify the selected MART
   org-unit level, so rows with empty/null hierarchy values were not excluded.
3. Boundary level inference shifted to the wrong level, so the map asked the
   staged boundary service for a level that did not exist in the local cached
   GeoJSON snapshot.
4. The staged boundary endpoint then attempted a live fallback, which made the
   map slow even though the correct local boundary snapshot data already
   existed.

The result was a rendered boundary layer with valid chart rows behind it, but
no value binding on the map because the renderer no longer knew which MART
column represented the displayed boundary level. In the failing case, the chart
returned district-level MART rows while the boundary request incorrectly asked
for level 4 geometry instead of the local staged level 3 district geometry.

## Fix

The compatibility path now infers hierarchy levels from the MART hierarchy
columns whenever `dhis2_ou_level` is missing. Explicit `dhis2_ou_level`
metadata still wins. When canonical legacy hierarchy columns are present, stray
mis-tagged helper columns are ignored instead of shifting the inferred level
numbers.

This keeps the data query on the normal MART chart-data flow while preserving
the previous fast local boundary-loading path. The fix restores:

- `dhis2_hierarchy_columns` serialization in Explore
- boundary-level to MART-column mapping in the frontend
- query-layer null exclusion for the selected org-unit level in the backend
- local staged GeoJSON boundary loading without synchronous live fallback during
  chart render

The inference only applies to legacy MART datasets with incomplete hierarchy
metadata. Datasets that already carry explicit `dhis2_ou_level` values keep
their stored levels.

For staged-local MART datasets, the backend now also excludes rows where the
selected org-unit hierarchy column is empty or null even when legacy hierarchy
metadata is incomplete. That keeps higher-level aggregate rows from polluting
map joins, groupings, labels, and tooltips.

## Compatibility Notes

- Legacy charts saved with the old default categorical map palette now fall
  back to sequential colors unless they use staged legends, manual breaks, or
  an explicitly selected non-default categorical palette.
- The shared `Data Range Color` control now resolves staged-local charts back
  to their original DHIS2 source database before loading legend sets. This
  prevents generic charts from querying the staged SQL database for DHIS2
  legend metadata.
- DHIS2 legend-set import in `Data Range Color` now uses an inline selector
  with swatch previews instead of a button-driven picker, matching the staged
  legend-loading approach already used by DHIS2 Map legend settings.
- Explore now merges shared palette, data-range, and chart background controls
  into a single `Color schemes` section for applicable charts instead of
  rendering separate color sections that overlap.
- DHIS2 Map now keeps one native `Color schemes` section for categorical vs
  sequential palettes plus staged/manual data-range coloring, and a separate
  `Map Style` section for viewport/background and boundary styling controls.
- A shared `Background color` control now applies behind the chart canvas, and
  DHIS2 Map also uses the same color inside the Leaflet viewport itself.
- The Labels section in Explore now opens expanded by default so label controls
  are immediately visible again.
- Boundary requests now stay on the staged local snapshot path. When a
  requested level is missing from the local snapshot, the service returns the
  local result immediately and schedules refresh work instead of blocking the
  chart render on a live fallback.
